description: 审查 brief/UI 设计稿的范围，决定是否需要拆分成多个 spec（Spec-Kit Step 0）

---

你正在执行 Spec-Kit 工作流的 **Prepare** 阶段，位于 specify 之前：**prepare** → specify → clarify → design → plan → tasks → analyze → finish。

## 目的

从源头控制颗粒度。一个 spec 太大 → Claude 一口气做完 → 中间没有验证节点 → 细节静默消失。Prepare 的唯一职责是：**读完输入后判断该不该拆、怎么拆**。

## 输入

- `$1`：feature 目录路径（必填）
- 自动读取（任一存在即可，按优先级）：
  1. `$1/brief.md` — 文字需求描述
  2. `$1/design.md` — 已抽象的设计索引
  3. `$1/design/` — 设计产物文件夹（截图 / HTML 原型 / Figma 链接等）
  4. `$1/finish.md` — 上一轮完成报告（如果是迭代）
- 项目原则：`constitution.md`（如存在则必读）

如果 `$1` 下既没有 brief.md 也没有 design.md 也没有 design/ 文件夹 → 报错：「请先在 `$1/` 下放置 brief.md 或设计产物，再运行 /prepare」并停止。

## 评估维度

读完所有输入后，评估以下 5 个维度：

### D1. 功能点计数
- 粗估 brief/设计稿中隐含的 FR（功能需求）数量
- 阈值：**≤ 15 FR → 合适**，**> 15 FR → 建议拆分**

### D2. 依赖层数
- 识别有几层依赖关系，例如：
  - 数据层（DB schema / migration）
  - API 层（新增或修改 route）
  - UI 层（新页面 / 组件）
  - 基础设施层（token 系统 / 认证 / 部署配置）
- 阈值：**≤ 2 层 → 合适**，**≥ 3 层 → 建议拆分**（每层应可独立验证）

### D3. 页面/模块数
- 涉及几个独立页面或模块
- 阈值：**≤ 3 个页面 → 合适**，**> 3 个页面 → 建议拆分**

### D4. 验证复杂度
- 做完后需要多长时间验证？需要检查几个页面、几个 API？
- 阈值：**2-3 分钟可验完 → 合适**，**需要逐页检查 > 5 分钟 → 建议拆分**

### D5. 前序状态
- 如果有 finish.md，检查上一轮遗留了什么
- 如果是全新功能，评估是否需要先打地基（token / schema）再做业务

## 输出格式

打印以下评估表给用户：

```
## Prepare 评估：<功能名称>

### 输入摘要
- brief.md：✅ / ❌（N 行，核心述求：一句话概括）
- design：✅ design.md / ✅ design/ (M 个文件) / ❌ 无
- finish.md：✅ / ❌
- constitution.md：✅ vX.X / ❌

### 范围评估

| 维度 | 估值 | 阈值 | 判定 |
|------|------|------|------|
| D1. FR 数量 | ~N 条 | ≤ 15 | ✅ / ⚠️ |
| D2. 依赖层数 | N 层（列举） | ≤ 2 | ✅ / ⚠️ |
| D3. 页面/模块数 | N 个（列举） | ≤ 3 | ✅ / ⚠️ |
| D4. 验证时间 | ~N 分钟 | ≤ 3 min | ✅ / ⚠️ |
| D5. 前序状态 | 描述 | — | ✅ / ⚠️ |

### 结论

**A) 不需要拆分** → 直接运行 `/specify $1`

或

**B) 建议拆分为 N 个子 spec**：

子 spec 目录创建在**父 spec 目录内部**，用字母后缀区分顺序：

| # | 子 spec 目录 | 范围 | 预估 FR | 依赖 |
|---|-------------|------|---------|------|
| 1 | `$1/XXXa-名称` | 描述 | ~N | 无 |
| 2 | `$1/XXXb-名称` | 描述 | ~N | 依赖 XXXa |
| 3 | `$1/XXXc-名称` | 描述 | ~N | 依赖 XXXb |

目录结构示例：
```
specs/001-feature/           ← 父 spec（保留 brief.md 作为整体方案参考）
  ├── brief.md               ← 父 brief（整体需求，不再单独 /specify）
  ├── 001a-基座/             ← 子 spec 1
  │   ├── brief.md
  │   ├── spec.md
  │   └── ...
  ├── 001b-核心逻辑/         ← 子 spec 2
  │   ├── brief.md
  │   └── ...
  └── 001c-验收/             ← 子 spec 3
      ├── brief.md
      └── ...
```

**执行顺序**：XXXa → finish 验证 → XXXb → finish 验证 → XXXc

**拆分原则**：
- 子 spec 嵌套在父 spec 目录下，父 brief.md 保留作为整体方案参考
- 每个子 spec 可独立部署、独立验证
- 有依赖关系的按层拆（地基先行）
- 每个子 spec 的 brief.md 需要用户确认后再 /specify
- 后续 /specify、/clarify、/plan 等命令传入子 spec 路径（如 `/specify $1/XXXa-名称`）
```

## 拆分操作

如果结论是 B（需要拆分）：

1. **不要自动创建子目录或 brief.md**——只给出建议
2. 列出每个子 spec 的建议 brief 要点（3-5 条），供用户参考
3. 用户确认拆分方案后，才为每个子目录创建 `brief.md` 骨架
4. 提示：「确认拆分方案后，我为每个子 spec 创建 brief.md 骨架，然后逐个运行 /specify」

如果结论是 A（不需要拆分）：
1. 直接提示：「范围合适，可以直接运行 `/specify $1`」

## 约束

- **只做评估，不做 specify**。不生成 spec.md、不写 FR、不做技术方案
- **不修改任何已有文件**
- 如果输入信息不足以判断（比如 brief 只有一句话），列出需要补充的信息，让用户补完再评估
- 设计产物（UI 截图 / 原型）和文字需求（brief）同等重要——都是功能描述，只是表达形式不同
